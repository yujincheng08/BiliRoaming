# AI 协作规范与开发指南

本文件旨在为所有介入 BiliRoaming 项目源码阅读、逆向分析和代码修改的 AI 或人类开发者提供全局的协作规范和指导原则。

## 1. 项目架构与逆向工程共识

BiliRoaming 是一个用于哔哩哔哩 Android 客户端的 Xposed 模块，核心技术栈为 **Kotlin + C++ (JNI)**。

### 🚨 核心原则：绝不允许硬编码
目标宿主（哔哩哔哩客户端）会频繁更新并混淆代码。**绝对禁止硬编码任何混淆后的类名、方法名或字段名。**

所有针对目标宿主的 Hook 和反射调用，必须依赖 `me.iacn.biliroaming.utils.DexHelper` 提供的底层 JNI 动态特征扫描机制：
- 通过独特的字符串常量匹配（`findMethodUsingString`）。
- 通过方法调用链/指令回溯匹配（`findMethodInvoked`, `findMethodInvoking` 等）。
- 所有的动态特征应配置并在运行时序列化到 Protobuf 缓存（`Configs.HookInfo`）中，以保证兼容性和冷启动性能。

## 2. 极其严格的代码修改纪律

在对本项目进行任何代码修改时，这是不可妥协的底线：

- **禁用自动格式化**：绝对禁止使用或触发任何代码自动格式化工具（如 Prettier, ktlint 的自动修复等）。
- **环境融合**：强制匹配目标文件周围现有的代码风格。这包括缩进层级、大括号放置习惯、空格以及变量命名约定。你的代码不应显得突兀。
- **最小化修改**：遵循“如无必要，勿增实体”的原则。仅修改实现当前业务逻辑所必需的最少代码行数，不要顺手“重构”或修改不相干的代码。
- **注释保留**：除非其描述的底层逻辑被彻底移除，否则必须完整保留现有代码中的所有注释。
- **修改前置条件**：在对任何现有文件进行修改（例如插入或替换代码）前，必须先执行 `cat` 或 `grep` 命令，精准确定目标代码的确切行号、上下文逻辑和当前的缩进级别，再进行精准的增删改。

## 3. 当前核心开发栈：SponsorBlock 集成 (恰饭片段跳过)

本项目当前正在推进的核心任务是集成 **SponsorBlock**，实现视频中的恰饭片段自动跳过。该任务的四大生命周期如下：

1. **API 异步对接**：
   - 与 SponsorBlock 的后端 API 进行交互，异步获取目标视频的片段跳过数据（如 `segment` 时间戳信息）。
2. **动态特征获取视频 CID/BVID**：
   - 寻找合适的切入点（例如 `BiliVideoDetail` 的解析或播放器初始化阶段），拦截并获取当前播放视频的唯一标识符（CID / BVID）。
3. **嗅探并 Hook 播放器进度回调**：
   - 动态定位 B 站播放器的核心服务类（如 `PlayerCoreService`），Hook 进度更新的回调方法或轮询进度机制。
4. **反射调用底层 `seekTo` 进行精准跳过**：
   - 当播放进度到达标记的 SponsorBlock 片段起始时间时，利用 `BiliBiliPackage` 中已缓存的 `seekTo` 方法引用，通过反射控制播放器精确跳转到片段结束时间。
